- 2、接着删除远程旧分支```git push origin --delete old - name```；
- 3、最后将重命名后的本地分支推送到远程```git push origin new - name``` 。

- 本地的push和merge会形成MERGE-HEAD(FETCH-HEAD)、HEAD（PUSH-HEAD）这样的引用。HEAD代表本地最近成功push后形成的引用。MERGE-HEAD表示本地成功pull后形成的引用。我么可以通过MERGE-HEAD或者HEAD，来实现与svn revet类似的效果。
- 解决将本地的冲突文件冲掉，不仅需要reset到MERGE-HEAD或者HEAD,还需要–hard。没有后面的hard，不会冲掉本地工作区。只会冲掉stage区。
`git reset --hard FETCH_HEAD`接着执行下面命令，就会成功。

### git版本回退
- ```git reflog``` 查看命令操作的历史
- ```git log```命令可以显示当前分支所有提交过的版本信息，不包括已经被删除的 commit 记录和reset的操作。(注意: 只是当前分支操作的信息)。
- ```git reflog```命令可以查看所有分支的所有操作记录信息（包括已经被删除的 commit 记录和 reset 的操作）。
- ```git reset --hard commit_id ```回退版本
- git reset 后, 本地版本回退了, 但无法直接 push 到远程仓库(因为远程仓库版本更加新) ```git push -f ```覆盖推送即可
- 回退前，用git log可以查看提交历史，以确定要回退的commit_id。
- 如果后悔回退了，用git reflog查看命令历史，以便确定要回到的版本的commit_id。

### git暂存
`git stash` 命令可以用于临时保存当前的修改，并将工作区回到上个提交状态，常用于需要切换分支或者处理紧急 Bug 的情况下。下面是 `git stash` 的具体使用方法：

- `git stash`：将当前所有未提交的修改存储到一个临时区域中，工作区恢复到上个提交状态。
- `git stash save "<message>"`：保存修改到临时区域，并添加一条保存说明。
- `git stash list`：列出所有已保存的临时修改区域。
- `git stash apply <stash>`：将指定的临时修改区域应用到当前分支上，但并不删除该临时修改区域。
- `git stash pop <stash>`：将指定的临时修改区域应用到当前分支上，并从列表中删除该临时修改区域。
- `git stash drop <stash>`：删除指定的临时修改区域。
- `git stash clear`：清空所有已保存的临时修改区域。

举个例子：假设当前在 `feature` 分支上开发新功能，但是需要立即切换到 `develop` 分支上处理 Bug，可以使用 `git stash` 保存当前的修改。具体操作如下：

```bash
$ git stash
Saved working directory and index state WIP on feature: 1234567 Add new feature
``` 

此时当前分支的工作区已回到上个提交状态，可以安全地切换分支处理 Bug。当需要恢复之前的修改时，可以使用 `git stash apply` 或 `git stash pop` 命令，例如：

```bash
$ git stash list
stash@{0}: WIP on feature: 1234567 Add new feature

$ git stash apply
``` 

这样，之前保存的修改便被应用到当前分支上了。如果需要删除一个特定的 stash，则可以使用 `git stash drop` 命令。

总之， `git stash` 命令可以帮助开发者快速保存工作进度，保持代码库的干净和整洁，同时也提高了在不同分支之间切换的效率。
### git refusing to merge unrelated histories报错
`refusing to merge unrelated histories` 错误通常是在合并两个不同仓库的历史时出现的错误。这个错误通常在使用 `git merge` 命令合并两个不同仓库的历史时出现。

出现这个错误的原因是，两个仓库的历史没有任何共同的祖先，因此 Git 不知道如何合并它们的历史。例如，如果你将两个不同的 Git 仓库合并为一个仓库，则会出现这个错误。

解决方法是在合并时添加 `--allow-unrelated-histories` 选项，告诉 Git 允许合并两个不同仓库的历史。你可以使用以下命令：

```
git merge --allow-unrelated-histories <branch-name>
```

其中 `<branch-name>` 表示你想要合并的分支名称。

需要注意的是，使用这个选项可能会导致某些冲突，因为两个不同的历史可能会有相同的文件名和行。你需要仔细处理这些冲突并解决它们。

另外，在 Git 2.9 或更高版本中，这个错误可能不会显示，默认情况下允许合并不相关的历史。如果你使用的是旧版本的 Git，则需要手动添加 `--allow-unrelated-histories` 选项来解决这个问题。

### 要暂存代码并创建一个新分支，然后将暂存的代码推送到新分支上，可以按照以下步骤进行操作：

- 使用 ```git status``` 命令检查当前的代码状态，确保没有未提交的更改或未跟踪的文件。

- 使用 ```git stash save``` 命令将当前的更改暂存起来。这将把当前的更改保存在一个临时的存储区域中，以便稍后恢复使用。

- 使用 ```git branch``` 命令创建一个新分支。例如，使用 git branch new-branch 创建一个名为 new-branch 的新分支。

- 使用 ```git checkout``` 命令切换到新创建的分支。例如，使用 git checkout new-branch 切换到 new-branch 分支。

- 使用 ```git stash apply``` 命令将之前暂存的更改应用到新分支上。这将恢复之前暂存的更改并将其应用到当前代码中。

- 使用 ```git add``` 命令将新分支上的更改添加到暂存区中。例如，使用 git add . 将所有更改添加到暂存区中。

- 使用 ```git commit -m "Commit message"``` 命令提交更改到新分支。请替换 "Commit message" 为适当的提交消息。

- 使用 ```git push origin new-branch``` 命令将新分支推送到远程仓库中。

- 现在，你已经成功地将暂存的更改应用到新分支上并将其推送到远程仓库。其他人可以基于这个新分支进行进一步的开发或审查你的更改。

# 常见的 Git 分支命名规范的建议：

## 主要分支：
- main 分支：主要用于发布稳定版本的代码。可以用其他名称如 master 替代，取决于团队的约定。
- develop 分支：用于整合各种特性分支，进行开发和集成测试。

## 特性分支：
- 特性分支用于开发新功能或解决特定问题。
- 命名规范：建议使用以下格式：
- ```feature/<feature-name>```：用于新功能开发，例如 ```feature/user-authentication```。
- ```bugfix/<bug-description>```：用于修复特定问题，例如 ```bugfix/fix-login-issue```。

## 发布分支：
- 发布分支用于准备发布版本。
- 命名规范：可以使用以下格式：
- ```release/<version-number>```：用于准备发布的版本，例如 ```release/1.0.0```。

## 热修复分支：
- 用于紧急修复生产环境中的问题。
- 命名规范：可以使用以下格式：
- ```hotfix/<issue-description>```：例如 ```hotfix/fix-security-issue```。

## 个人分支：
- 每个开发人员可以创建自己的个人分支来进行开发工作。
- 命名规范：可以使用以下格式：
- ```user/<username>/<feature-name>```：例如 ```user/johndoe/new-feature```。

## 临时分支：
- 有时需要创建临时性的分支来进行实验、测试或快速修复。
- 命名规范：可以使用以下格式：
- ```temp/<description>```：例如 ```temp/experiment-with-new-feature```。

## 任务分支：
- 有些团队根据任务或问题创建分支。
- 命名规范：可以使用以下格式：
- ```task/<task-id>/<description>```：例如 ```task/123/implement-search-feature```。

## 版本标签：
- 版本标签用于标识发布的版本号。
- 命名规范：可以使用语义版本号格式，例如 v1.0.0。
- 确保团队成员都理解并遵守这些规范。
